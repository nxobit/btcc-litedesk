# 本地钱包 PBE 加密升级说明

本文说明 BTCC 钱包本地敏感数据加密从旧格式升级到新格式的原因、兼容策略和实际行为。

## 目标

本地数据库里保存了加密后的助记词和 WIF 私钥。旧实现使用：

- `PBKDF2-HMAC-SHA512`
- `AES-256-CBC`
- 迭代次数 `1000`

`1000` 次对于本地桌面钱包偏低，数据库一旦泄漏，弱密码的离线爆破成本不够高。当前实现已将新写入密文升级为：

- `PBKDF2-HMAC-SHA512`
- `AES-256-CBC`
- 迭代次数 `20000`

## 新旧格式

旧格式没有版本头，密文直接是 Base64 文本，解密时默认按 `1000` 次处理。

新格式在密文前增加了版本和参数头，格式为：

```text
v2$20000$<base64-payload>
```

含义如下：

- `v2`：当前密文版本
- `20000`：本条密文实际使用的 PBKDF2 迭代次数
- `<base64-payload>`：`salt + iv + ciphertext` 的 Base64 编码

这样后续如果还要继续提高参数，不需要再猜测旧数据是按什么规则加密的。

## 兼容策略

本次升级不破坏旧数据库。

兼容规则如下：

1. 读到 `v2$...` 头时，按头里的迭代次数解密
2. 没有版本头时，视为历史旧密文，按 `1000` 次解密

也就是说：

- 旧库仍然可以正常打开
- 原有钱包密码不需要修改
- 用户不需要导出、重建或重新导入钱包

## 静默迁移

升级不是靠数据库 schema 迁移完成的，而是在用户已经提供正确钱包密码后，程序顺手把旧密文重写成新格式。

当前有两条迁移路径：

1. 钱包列表页解锁成功后，批量扫描旧密文并重写为 `v2$20000$...`
2. 某条钱包在解密成功时，如果发现还是旧格式，也会立刻按新格式覆盖回数据库

这样做的结果是：

- 第一次成功解锁后，大部分旧密文会被静默升级
- 第二次开始，系统看到 `v2$20000$...`，就直接按新参数解密

## 对现有用户的影响

这次升级对现有用户的影响应当是无感的：

- 钱包密码不变
- 钱包地址不变
- 助记词和私钥内容不变
- 只改变数据库内密文的封装格式和 PBKDF2 迭代次数

需要注意的是，这次升级提升的是“数据库泄漏后的离线爆破成本”，不是替代系统层面的设备安全、权限隔离或助记词备份安全。
